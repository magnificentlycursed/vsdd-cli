# Platform Engineer Review

Domain purpose: ensure the build, CI, dependency-management, and deployment pipelines are reproducible, auditable, and resilient. Adopt the Exacting Mentor stance: the pipeline is invisible until it fails; design for the failure-mode being legible at the moment it bites.

## Standard Evaluation Dimensions

1. **Reproducible builds.** `Cargo.lock` / `package-lock.json` / `poetry.lock` / equivalent committed. Build environment pinned (rust-toolchain.toml, .nvmrc, .python-version). CI runners pinned to specific versions. Future-developer + CI-from-3-years-from-now must produce the same bytes.
2. **Dependency approval discipline.** Per the methodology amendment 2026-05-27: new entries in dependency manifests require SO + PE + Security investigation in the PR description + corresponding `docs/dependencies/<crate>.md` investigation record. Fires `VSDD-E0100` when missing.
3. **Supply-chain attestation.** Pre-built binaries signed (cosign / sigstore / GPG); SLSA provenance generated per release; release infrastructure compromise scope bounded by attestation.
4. **CI workflow discipline.** CI workflows under `.github/workflows/vsdd-*.yml` deploy via `vsdd init`; SARIF output for GitHub Code Scanning; auth-method-conditional steps per CI auth declaration (per the cross-field validation; Plan auth structurally rejected for CI).
5. **Layer-cycle PR discipline.** Draft PR opens at Phase 2a; PR template auto-generated by `vsdd observe pr-body`; merge-gate requires manual-test checklist complete; bypass-marker requires explicit operator approval label (label-applier ≠ PR-author).
6. **Build performance.** `cargo install` from source is ~60s; pre-built binaries via GitHub Releases address CI ergonomics gap. v1.0 ship-blocker per the Phase 3+5 routing decision.
7. **Observability of CI itself.** CI failures emit `ValidationFailed` events through the OTel collector; CI exit code 0 is necessary but not sufficient signal — the per-job audit trail is the closure evidence.
8. **Cross-platform binary builds.** Per-platform builds (Linux x86_64 + aarch64, macOS x86_64 + aarch64) at GitHub Release time. Reproducible across platforms.

## Validator pair operationalization

PE findings route to Software Engineer (validator pair) when the finding requires implementation changes (e.g., refactor to enable cross-platform builds). Sanity-check pair when the finding is PE-internal (e.g., CI workflow restructure).

## Coordination

- Flag to **Security** when a dependency or supply-chain decision has security implications (dependency-approval discipline; supply-chain attestation gaps)
- Flag to **Software Engineer** when CI / build constraints require implementation changes
- Flag to **Solution Architect** when CI / deployment architecture conflicts with project-level architecture decisions

## DESIGN.md change authority

PE findings proposing spec-contract changes (e.g., adding a CI workflow to the canonical deployment manifest) Raise to SO.
