# Security Review 1 — 2026-05-27

**Phase 5 surface:** C — Fuzz Testing analog (adversarial-input methodology fuzz; the methodology is the parser, adversarial-project edge cases are the inputs)
**Cold-session shape:** N/A — inline-run from the main session. Trade-off declared per the bounded-judgment-surface rubric: adversarial-input fuzz is a finite enumeration of named edge cases (zero-axes; non-GitHub; init-order; auth × CI; identity continuity; methodology drift) traced through the methodology spec. Cold-session cluster spawn would be over-investment for the bounded enumeration.

## Scope

The methodology spec (DESIGN docs + README) describes the canonical project shape: GitHub-hosted, crosslink-then-vsdd init order, Plan-auth-for-operator-local + API-key-for-CI, single operator-identity per project, one canonical methodology.md per toolkit version. Adversarial-input fuzz feeds projects that violate each of those assumptions and traces what the methodology does:

1. Zero-axes project (all 9 axes declared `no`)
2. Non-GitHub platform (GitLab / Bitbucket / Forgejo / Codeberg / self-hosted Gitea)
3. Reversed init order (vsdd init before crosslink init)
4. Plan-auth-everywhere project (CI also tries to use Plan auth)
5. Multi-machine operator (laptop + desktop + work + personal)
6. Methodology drift (adopting project's methodology.md vs vsdd-cli's canonical methodology.md after toolkit upgrade)

For each, trace what the methodology asserts and find the assumption-violation surface.

## Findings

### Finding 1 — Zero-axes project edge state is undefined (Dim: input-validation completeness) — Open

A project that declares all 9 per-feature axes as `no` in `.vsdd/config.yaml` has no axis-driven domain activations. Per the README axis → activation matrix, no domain activates. Phase 3 composed_domains would be `[]`. Cluster-batching (which presumes 4 clusters × multi-domain) collapses. Phase 5 (which composes QE + Security + SA per DESIGN-METHODOLOGY's phase-domain matrix) — wait, the phase-domain matrix says Phase 5 composes QE + Security + SA *regardless of axes*. So Phase 5 has hard-coded composition; Phase 3 doesn't.

**The fuzz signal:** the methodology has two distinct composition mechanisms — phase-fixed (Phase 5 hard-codes QE + Security + SA) and axes-driven (Phase 3 reads axes). The two mechanisms are not unified; an operator reading the methodology cannot derive a single rule "for any phase + project, the composed domains are X." Phase 3 is the only phase that depends on axes; every other phase has hard-coded composition that ignores axes. Zero-axes project completes phases 1a-2c + 5 + 6 with the hard-coded compositions but has Phase 3 in undefined state.

**Why this matters:** the methodology spec presents the axes matrix as a load-bearing decision-driver, but it only drives Phase 3. The "no axes activated → no domains" path silently degrades Phase 3 to a no-op rather than naming the degraded state.

**Routing:** Phase 4 → Phase 1a (DESIGN-METHODOLOGY + README coordinate revision). Composes with QE's Finding 4 (implicit always-on baseline). Operator-directive needed to pick the methodology floor.

### Finding 2 — GitHub-platform assumption baked into adoption flow (Dim: portability / threat-model assumption) — Open

`vsdd init` deploys `.github/PULL_REQUEST_TEMPLATE.md`, `.github/CODEOWNERS`, `.github/workflows/vsdd-*.yml`, `.github/ISSUE_TEMPLATE/*.md`. The bypass-marker enforcement (CI label-applier ≠ PR-author check, `DESIGN-VERIFICATION.md:363-394`) is GitHub-API-specific. The dependency-approval discipline (just landed in commit a0a4987) extends the same GitHub assumption. The README's PR template + Layer-cycle PR discipline + CHANGELOG-cooperation-with-crosslink-close pattern presume GitHub semantics.

**The fuzz signal:** a project hosted on GitLab / Bitbucket / Forgejo / Codeberg / self-hosted Gitea / sourcehut would either silently fail the deployment (no `.github/` directory; templates land but never trigger) OR refuse to deploy entirely. The methodology has no platform-detection or alternative-platform fallback.

**Why this matters:** vsdd-cli claims to compose against "Claude Code substrate + crosslink" but in practice composes against "Claude Code substrate + crosslink + GitHub." The CHANGELOG-cooperation pattern, the bypass-approval label gate, the CODEOWNERS auto-routing, the SARIF emission for GitHub Code Scanning — all assume GitHub. An adopting project on a non-GitHub platform loses substantial portions of the verification subsystem's CI-side teeth.

**Routing:** Phase 4 → Phase 1a (DESIGN-METHODOLOGY + DESIGN-VERIFICATION + README coordinate revision). Three options:

- (a) Declare the methodology GitHub-only at v1; v1+ adds GitLab / Forgejo support (named scope expansion).
- (b) Abstract the platform-specific surfaces (CI label gate, CODEOWNERS, SARIF) into pluggable adapters; v1 ships GitHub adapter; v1+ adds others.
- (c) Document each GitHub-specific surface with its non-GitHub fallback (e.g., "for GitLab: use approval rules instead of bypass-approved label").

Operator-directive needed.

### Finding 3 — Init-order dependency is not declared (Dim: operator-experience predictability) — Open

The README's adoption flow (`cargo install vsdd && crosslink init && vsdd init`) presents `crosslink init` before `vsdd init` as the canonical order. The collision-handling matrix (README.md:172-181) implies the order is `vsdd init` reading crosslink's already-deployed files (`.claude/mcp.json` already has crosslink entries; vsdd adds its own).

**The fuzz signal:** an operator who runs `vsdd init` first then `crosslink init` later gets:
- `vsdd init` writes `.claude/mcp.json` (creating it) with only the vsdd entry
- `crosslink init` then runs JSON object merge against the existing file — but crosslink expects to be the first writer; behavior in this collision is unspecified
- Both tools' init-manifest tracking is independent; neither knows the other ran in unexpected order

Reading the methodology, an operator cannot predict whether the reverse order works. The collision-handling logic suggests it should be order-independent, but the spec doesn't explicitly state so.

**Why this matters:** init-order assumptions are silent and discovered only at operator-first-use. Either we guarantee order-independence (with test fixtures proving it) or we name the required order with a `vsdd init --check` pre-flight failure when crosslink-init-manifest absent.

**Routing:** Phase 4 → Phase 1a (DESIGN-METHODOLOGY + README revision). Decision: `vsdd init --check` detects crosslink-absent and either prompts to run `crosslink init` first OR documents the reversed-order semantics. Lower-priority than Findings 1+2 — operator can probably empirically discover the right order; doesn't block adoption.

### Finding 4 — Plan auth + CI is structurally permitted but operationally impossible (Dim: input-validation cross-field) — Open

The `.vsdd/config.yaml` schema permits `auth_method: plan, auth_method_credential_source: plan-auth-no-key`. The schema validator passes the declaration. But Plan auth requires an operator-interactive session that CI cannot provide — CI runners can't authenticate against Anthropic's Plan tier without an operator browser session.

**The fuzz signal:** a project declares `auth_method: plan` in config + deploys CI workflows via `vsdd init`. The schema validator + the methodology hooks all pass. The CI run fails at runtime when the Agent SDK cannot authenticate (no credential resolves to `plan-auth-no-key`). The methodology has no cross-field validation that rejects `auth_method: plan` when CI workflows are present.

**Why this matters:** auth-method × deployment-surface compatibility is a structural property. The schema validator declines to enforce it; the CI workflows trust the config; the runtime is the first surface that catches the impossibility. Phase 5 should surface this as a hard constraint, not a runtime surprise.

**Routing:** Phase 4 → Phase 1a (DESIGN-METHODOLOGY auth-method + DESIGN-SCHEMA cross-field-validation revision). Three options:

- (a) Separate `auth_method.operator_local` + `auth_method.ci` fields; cross-field validation rejects `ci: plan`.
- (b) Single `auth_method` field with `ci_compatible: bool` deriving from auth-method choice; methodology rejects deploying CI workflows when `ci_compatible: false`.
- (c) Schema-level enum constraint: `auth_method: api_key` required if any CI workflow file exists in `.github/workflows/`.

Suggest (a) — most operator-honest representation; matches how operators think about auth-per-context.

### Finding 5 — Multi-machine operator identity continuity is unaddressed (Dim: identity / attribution discipline) — Open

The methodology's per-operator vs shared-organizational auth_attribution_pattern (README.md:275) assumes consistent operator identity across cycles. But an operator working from multiple machines (laptop + desktop + work + personal) has:
- Different SSH key fingerprints per machine (the Exit Signal attestation's signature field would not be consistent)
- Potentially different git user.email per machine (the Commit-level domain co-authorship trailers would have variable attribution)
- Potentially different `auth_method_credential_source` env-var names per machine

The methodology defaults to per-operator attribution but doesn't define "per-operator" as "per-identity-fingerprint-set" vs "per-account."

**The fuzz signal:** an operator who works the same project from 2 machines emits divergent identity signatures in the event log. The audit trail shows two distinct attestation identities for what is "one operator." Per the "Phase 6 Exit Signal" attestation discipline (signed by attested_by's registered key), the operator must declare which SSH key fingerprint is canonical for the attestation — but the methodology doesn't address what happens if the registered key rotates between cycles or if multiple operator-keys are valid.

**Why this matters:** Goal 4 (CI/CD + audit-trail integrity) depends on consistent attribution. Multi-machine operators are common; the methodology silence is a real-world gap.

**Routing:** Phase 4 → Phase 1a (DESIGN-METHODOLOGY auth + identity discipline revision). Options:

- (a) Operator-identity-as-set-of-fingerprints: `.vsdd/config.yaml` carries `operator_keys: [fingerprint-1, fingerprint-2, ...]`; any registered fingerprint can sign attestations; methodology accepts the union.
- (b) Single canonical operator key with rotation events: methodology requires one key per project; rotation emits `OperatorDirectiveApplied{directive: operator-key-rotation, from_fingerprint, to_fingerprint}`.
- (c) Defer to v1+ pending evidence of recurrence (single-operator projects don't hit this case in v1 evaluation cycles).

Suggest (c) for v1; promote when adoption surfaces evidence.

### Finding 6 — Methodology spec drift between toolkit-canonical and project-local copies (Dim: cross-source / supply-chain consistency) — Open

The methodology declares (DESIGN-METHODOLOGY.md:51): "Lives at `methodology.md` at project root (for vsdd-using-projects) and at `vsdd-cli` repo root (for the toolkit's own canonical spec)." `vsdd init` presumably deploys the canonical methodology.md to the adopting project at adoption time.

**The fuzz signal:** the operator runs `cargo install vsdd --force` to upgrade to vsdd-cli v1.2 from v1.0. The toolkit's canonical methodology.md now declares (hypothetically) `methodology_version: 1.2.0` with new disciplines added in v1.1 and v1.2. The adopting project's methodology.md is still the v1.0 copy deployed at first `vsdd init`. The validator hooks bundled in vsdd v1.2 enforce v1.2 disciplines against an adopting-project methodology.md that asserts v1.0 discipline. Validator output references rules the project's local spec doesn't acknowledge.

The methodology has no version-pinning mechanism between toolkit and project; no `vsdd init --update` to refresh the methodology.md; no managed-section pattern for methodology.md like CHANGELOG has. The methodology.md file is "deployed once and forgotten."

**Why this matters:** the dual-source pattern (canonical + project-local) is supply-chain-risky. An adopting project may be running v1.0 methodology while the toolkit is v1.2; operator-facing errors reference v1.2 rules; methodology spec on disk says v1.0. The discrepancy is exactly the kind of audit-trail-corruption the suite-development reviews surfaced as `document-staleness` patterns in the existing suite.

**Routing:** Phase 4 → Phase 1a (DESIGN-METHODOLOGY methodology-spec discipline revision). Three options:

- (a) Version-pin: methodology.md frontmatter carries `methodology_version: <semver>`; vsdd verify check warns when project methodology_version < toolkit methodology_version; `vsdd init --update-methodology` refreshes.
- (b) Reference-by-URL: project methodology.md is a one-liner `See https://github.com/magnificentlycursed/vsdd-cli/blob/v1.2.0/methodology.md`; no local copy; always reads canonical.
- (c) Managed-section pattern: methodology.md sections marked `<!-- === vsdd managed === -->` get auto-updated on `vsdd init`; operator-customizations outside markers preserved.

Suggest (a) — preserves operator-local-spec authority while enabling drift detection.
