# Security Review 3 — Crosslink identity-leak incident — 2026-05-28

## TL;DR

The `crosslink/hub` branch was pushed publicly to `github.com/magnificentlycursed/vsdd-cli` via crosslink's default `tracker_remote` behavior (defaults to `origin` when unset). The pushed `trust/keys/inhz.pub` file contained the operator's SSH public key with comment `crosslink-agent:inhz@<FirstName>s-MacBook-Pro.local` — the hostname leaked the operator's first name (the macOS default hostname format `<FirstName>s-MacBook-Pro.local` embeds the user's first name as configured at OS install time), weakening the GitHub-handle-per-`github.com/magnificentlycursed/`-URL pseudonymous facade.

The leak occurred during a scripted issue-filing session (89+ `crosslink issue create` / `comment` / `close` / `label` invocations) in which the author dismissed the recurring `WARN no tracker_remote configured ... defaulting to "origin"` message by suppressing stderr (`2>/dev/null`) as a workaround for an unrelated bash variable-capture bug. The WARN was the signal that auto-push was happening; suppressing it removed the signal without addressing the underlying behavior.

After Option 1 remediation (commit `5c7543a` on `main`) landed and `main` was pushed publicly for the first time, a second, distinct leak was surfaced by the operator: `.crosslink/driver-key.pub` is a tracked file (not gitignored) committed in the initial `crosslink init: deploy substrate` commit (`b3d6e50`), containing an SSH public key whose comment references a different pseudonymous persona — one not associated with this project. Remediation of this second leak is deferred pending operator audit of cross-persona key-publication surface.

## Second leak — `.crosslink/driver-key.pub` (remediation deferred)

After Option 1 landed, the operator surfaced a second, distinct identity leak via direct repo observation: the `.crosslink/driver-key.pub` file is tracked (intentionally per crosslink's design — driver-key is meant to be shareable for trust verification) and contains an SSH public key whose comment references a pseudonymous persona unrelated to this project.

This is a **different failure mode** than the first:
- The agent-key leak (`trust/keys/inhz.pub` on `crosslink/hub` branch) was a **generated-identity leak** — crosslink generated a new keypair at agent init time + auto-derived the SSH key comment from system hostname + auto-pushed to the public hub branch
- The driver-key leak (`.crosslink/driver-key.pub` on the `main` branch) is a **sourced-identity leak** — crosslink-init sourced an existing SSH key (mechanism to confirm with upstream maintainer; likely `~/.ssh/id_ed25519.pub` or similar) and copied it into the project repo at the tracked `.crosslink/driver-key.pub` path **including the original key's existing comment**, without scrubbing or warning

**Scope of the second leak:**
- File: `.crosslink/driver-key.pub` (103 bytes; ssh-ed25519; 22-char comment containing a pseudonymous identifier)
- Tracking: TRACKED (intentional per crosslink design)
- First committed: `b3d6e50` (initial crosslink-init substrate deployment, days before this session)
- Public state: present on `main` branch at `github.com/magnificentlycursed/vsdd-cli/blob/main/.crosslink/driver-key.pub`; republished publicly when `main` was first pushed during Option 1 step (b.1)
- Linkage to operator's other personas: the comment contains an identifier tied to a different pseudonymous persona; the cryptographic key bytes themselves may also be linkable to that persona via any other repository where the same key is published (cross-repo cryptographic linkage)

**The wrinkle that distinguishes this leak from the first:** for the agent-key leak, redacting only the SSH key comment was sufficient because the cryptographic key bytes were generated fresh by `crosslink init` for this project specifically — the fingerprint wasn't linkable to other contexts. For the driver-key leak, the cryptographic key bytes themselves may be linked to other publications of the same key in other pseudonymous projects. Comment redaction alone may not eliminate the cross-persona linkage.

**Status:** **deferred** per operator-directive 2026-05-28. Operator is auditing the cross-persona key-publication surface before choosing a remediation path. The leak persists on public `main` until the audit completes.

## Reproduction (for upstream filing)

**Environment:**
- crosslink version: per `which crosslink` → `/usr/local/bin/crosslink` (installed via cargo)
- Operating system: macOS (default `hostname` of form `<FirstName>s-MacBook-Pro.local`)
- Git remote: standard `origin` pointing to a public GitHub repository
- Session shell: bash (zsh-default macOS)

### Failure mode 1 — Generated identity leak (agent key)

**Reproduction steps:**

1. `crosslink init` in a new repository → generates `.crosslink/agent.json` with `machine_id` set to system `hostname` output (e.g., `<FirstName>s-MacBook-Pro.local`)
2. `crosslink init` generates an SSH keypair at `.crosslink/keys/<agent_id>_ed25519` with public-key comment formatted as `crosslink-agent:<agent_id>@<machine_id>`
3. The public key file is published to the `trust/keys/<agent_id>.pub` path on the `crosslink/hub` branch on first crosslink mutation operation
4. With `tracker_remote` unset in `.crosslink/hook-config.json` (the default after `crosslink init`), the `crosslink/hub` branch is auto-pushed to `origin` on every mutation operation
5. Console prints `WARN no tracker_remote configured in <abs-path>/.crosslink/hook-config.json, defaulting to "origin"` to stderr on every operation (the WARN message also leaks the operator's absolute home path in its own text)
6. After any `crosslink issue create` (or comment / close / label / etc.), `git ls-remote --heads <origin-url>` shows `refs/heads/crosslink/hub` is present + the GitHub repo's `trust/keys/<agent_id>.pub` file is visible at `https://github.com/<owner>/<repo>/blob/crosslink/hub/trust/keys/<agent_id>.pub`

**Observed leaked content on `crosslink/hub`:**

```
ssh-ed25519 <40-char-key-bytes-redacted> crosslink-agent:inhz@<FirstName>s-MacBook-Pro.local
```

The first part (`ssh-ed25519 AAAA...`) is the public-key bytes (publishing those is the intended trust mechanism). The trailing comment (`crosslink-agent:inhz@<FirstName>s-MacBook-Pro.local`) is where the hostname leaks — and where it carries the operator's first name on the macOS default hostname pattern.

**Other identity content propagated to `crosslink/hub`:**
- Every `issues/<uuid>/issue.json` file contains `created_by: "inhz"` (pseudonymous agent ID — OK by itself, but cross-referenceable via the SSH key comment)
- Every `issues/<uuid>/comments/<uuid>.json` contains `agent_id: "inhz"` (same)
- Commit metadata on the hub branch is authored by `<github-id>+<github-handle>@users.noreply.github.com` (the GitHub noreply email per `noreply.github.com` — already public by virtue of being the repo's owner identity)
- Commit subject lines have the form `inhz: create issue: <title> at <timestamp>`

### Failure mode 2 — Sourced identity leak (driver key)

**Reproduction steps:**

1. `crosslink init` in a new repository → sources an existing SSH key (likely from `~/.ssh/id_ed25519.pub` or asks operator to provide one; exact mechanism to confirm with maintainer)
2. The sourced key is copied verbatim — including its existing comment — to `.crosslink/driver-key.pub` (a TRACKED file, not gitignored)
3. On any commit + push of the project's `main` (or default) branch, the file becomes publicly visible at `https://github.com/<owner>/<repo>/blob/<default-branch>/.crosslink/driver-key.pub`
4. No warning is printed about the comment portion of the sourced key being copied into a tracked file

**Observed leak surface:**
- Scope: 1 file (`.crosslink/driver-key.pub`); 103 bytes; ssh-ed25519 format; 22-char comment containing an identifying string tied to another pseudonymous context
- Cross-reference linkage: the cryptographic key bytes themselves may match a key published in another repository tied to a different pseudonymous identity; this links the two pseudonymous projects cryptographically regardless of comment content

### Expected vs actual behavior (the crosslink design assumption)

Crosslink's design optimizes for the **transparent-team-collaboration use case**: multiple agents on multiple machines need to verify which agent's commits came from which machine. The SSH key comment with `<agent_id>@<machine_id>` is a human-readable identity affordance for that use case. The auto-push of hub to origin is the cross-machine sync mechanism. The tracked driver-key.pub is the trust-anchor sharing mechanism.

For **pseudonymous solo-operator projects** (this repo's case: the operator's GitHub identity per `github.com/magnificentlycursed/` is intentionally pseudonymous; the `noreply.github.com` email is the GitHub noreply form), the defaults leak identity:
- `machine_id` defaults to `hostname` output, which on macOS contains the user's first name
- `tracker_remote` defaults to `origin`, which is typically the public GitHub remote
- Auto-push is enabled by default; no opt-out signal beyond the suppressible WARN
- Sourced driver-key is copied with its original comment intact
- Cross-context cryptographic linkage via shared key bytes is not warned about

The mismatch: the pseudonymity-conscious operator gets no actionable indication that the WARN-suppressible default is publishing identifying content publicly.

## Defects revealed (security-domain class)

### Defect 1: Anonymization registry coverage gap (closed by commit `5c7543a`)

**State before incident:** `templates/registry/anonymization-patterns.yaml` covered API key patterns + env-var-credential patterns + `$HOME-path` + `git-user-name` + `git-user-email`.

**Missing patterns that the incident surfaced:**
- Machine hostnames (e.g., `<FirstName>s-MacBook-Pro.local` — leaks first name on macOS default form)
- SSH public key comments with host info (e.g., `crosslink-agent:<id>@<host>` form crosslink generates)
- Absolute user paths even when `$HOME` env var is not exported in the hook context (e.g., `/Users/<username>/...` patterns referenced in non-shell tool output)
- The crosslink-specific `WARN no tracker_remote configured` operator-runbook signal

**Status:** closed by commit `5c7543a`'s registry extension. The legacy `check-anonymization.sh` hook (installed at `.git/hooks/pre-commit`) covers the `$HOME` case immediately; the four new patterns added to the registry will activate when the toolkit's `vsdd init` deploys the registry-aware hook (pending implementation per the methodology's track 2b).

### Defect 2: Methodology specifies `check-anonymization` hook but no implementation deployed

**State:** the methodology's anonymization discipline is spec'd but no hook is actually deployed on the vsdd-cli repo. The legacy `vsdd-suite/hooks/check-anonymization.sh` exists in the existing-suite but was not installed on vsdd-cli (this is a spec-stage repo; the methodology's hooks deploy via `vsdd init` which doesn't exist as a built binary yet).

**Planned mitigation:** Layer 3 of the toolkit's `DESIGN.md` (`vsdd verify` + ~19 methodology hooks) will implement + deploy the anonymization hook. Until then, the local pre-commit shim installed in Option 1 step (e) provides interim protection.

**Status:** mitigation planned; tracked by Layer 3 acceptance criteria in `DESIGN.md`.

### Defect 3: Methodology's check-anonymization scope doesn't cover git-plumbing operations

**Class:** mechanical-enforcement gap.

**State:** the legacy `check-anonymization.sh` fires on `pre-commit` for staged files. Crosslink writes the hub branch via direct git plumbing (`git update-ref` + `git commit-tree` on a separate ref) that **bypasses pre-commit hooks entirely**. Even if the hook had been installed, it would not have caught the leak because crosslink's hub mutations don't go through `git commit`.

**Planned mitigation:** none yet spec'd. The methodology should add either:
- A `check-crosslink-hub-anonymization.py` hook that fires on `pre-push` (catches before publication, not at commit time)
- OR an integration with crosslink to register the anonymization check as a hub-mutation gate

**Status:** **open**. Recommended methodology amendment: extend the check-anonymization discipline to cover `pre-push` not just `pre-commit`; OR define a crosslink-hub-mutation-gate integration. This amendment routes to the VSDD Methodology meta-domain (see corresponding methodology review for this date).

## Upstream filing — Crosslink usability + safety feedback

The following is intended to be filed as an upstream issue at the crosslink repository (`github.com/forecast-bio/crosslink` per the substrate composition declaration). Cut-paste from this section is OK.

### Generated vs Sourced identity leak — two distinct failure modes

This incident surfaced **two distinct identity-leak failure modes** in crosslink's default behavior. Both follow the same root design assumption (transparent-team-collaboration default) but require different remediation. Filing them as a single issue (with sub-cases) or as separate-but-linked issues is the upstream maintainer's call.

**Failure mode 1: Generated identity leak (agent key)**
- Mechanism: `crosslink init` generates a fresh ed25519 keypair for the agent role; sets the SSH key comment template to `crosslink-agent:<agent_id>@<machine_id>` where `machine_id` is derived from system `hostname` output
- Pre-leak state: no identity content existed for this project
- Leak state: project-specific identity content (the hostname) is published to `crosslink/hub` (publicly via auto-push when `tracker_remote` defaults to `origin`)
- Catchable at: agent-init time (sanitize the hostname embedded in the SSH comment template)
- Remediation: rotate machine_id to anonymized value; rotate SSH key comment; cryptographic key bytes can stay (no cross-context linkage since key was generated for this project)

**Failure mode 2: Sourced identity leak (driver key)**
- Mechanism: `crosslink init` sources an existing SSH key (mechanism to confirm with maintainer; likely `~/.ssh/id_ed25519.pub` or similar) and copies it into the project at the tracked `.crosslink/driver-key.pub` path **including its existing comment**, without scrubbing or warning
- Pre-leak state: an SSH key existed at the source location with identifying comment + cryptographic bytes potentially used in other published contexts
- Leak state: the entire key file (key bytes + comment) is published to `main` branch (tracked, default-pushed)
- Catchable at: source-key-copy time (warn that the sourced key's comment contains potentially-identifying content; offer to anonymize the copy)
- Remediation: more complex; comment redaction alone may not eliminate cross-context linkage; operator may need to rotate to a fresh keypair specifically for this project

### Upstream issue draft

**Title:** Default behavior leaks identity content (auto-push of `crosslink/hub` + sourced-key copy into `.crosslink/driver-key.pub`) for pseudonymous solo-operator projects

**Type:** usability / safety

**Severity:** medium (working as designed for primary use case; significant identity exposure for pseudonymous use case; signal-of-mechanism present but easily suppressed)

**Affected versions:** crosslink as of 2026-05-28 (the version installed locally; check `crosslink --version` for the specific build). The behavior may be present in all versions; the upstream issue should request maintainer-side confirmation.

**Proposed fixes (suggestions for crosslink maintainer to consider):**

1. **First-time-use prompt:** on first mutation operation with `tracker_remote` unset, interactively prompt the operator (TTY-aware) with: "Crosslink's `tracker_remote` is unset; about to auto-push hub branch to `origin` (which appears to be `<remote-url>`). For pseudonymous projects you may want `crosslink config set tracker_remote local-only` instead. Continue? [y/N/local-only]". For non-TTY contexts (CI; scripted use), require explicit `tracker_remote` value OR fail with clear ERROR.

2. **Clearer WARN message:** if the WARN approach is retained, change the message text to: `WARN tracker_remote unset; about to auto-push 'crosslink/hub' to '<resolved-remote-url>' on this and future operations. Run 'crosslink config set tracker_remote local-only' to disable auto-push, or 'crosslink config set tracker_remote origin' to acknowledge.`

3. **Suppress home-path in WARN text:** the WARN currently includes the absolute path of `hook-config.json` which leaks the operator's home directory. Use a path relative to the repo root + `<repo-root>` placeholder.

4. **Optional machine_id override at agent init:** `crosslink agent init --machine-id <custom>` should accept an explicit anonymized value (`crosslink agent init --machine-id redacted` or `crosslink agent init --anonymous` as a convenience flag). Currently the machine_id is auto-derived from `hostname` with no override surface.

5. **Pseudonymity workflow documentation:** the crosslink-contract.md (referenced from the existing vsdd-suite at `vsdd-suite/crosslink-contract.md`) should add a section "Pseudonymous solo-operator workflow" naming the recommended `tracker_remote: local-only` + `machine_id: <custom>` settings + the rationale.

6. **Driver-key sourcing transparency:** when `crosslink init` sources an existing SSH key to populate `.crosslink/driver-key.pub`, the operator should be shown the source path + the comment portion that will be copied into the project's tracked file. Specifically:
   - Print: "About to copy SSH key from `<source-path>` to `.crosslink/driver-key.pub` (tracked file; visible to other developers of this repo). Key comment: `<comment>`. Continue? [y/N/anonymize]"
   - The `anonymize` option strips the comment before copying
   - For non-TTY contexts: require explicit `--driver-key <path>` + `--driver-key-comment <comment>` flags; fail without them

7. **Driver-key cross-context linkage warning:** crosslink should detect when the sourced driver-key's public bytes already exist in another tracked repository on the same machine (via fingerprint comparison against other `crosslink/`-managed repos OR a scan of `~/.ssh/*.pub`). When detected, warn: "Driver-key with fingerprint `<sha256>` is already used in `<other-repo-path>`. Publishing this key here links the two repositories cryptographically. Consider generating a fresh driver-key for this project."

**Workaround for current users (verified working on this repo as of 2026-05-28):**

```sh
# Disable hub auto-push (per-repo; stored in .crosslink/hook-config.local.json which is gitignored)
crosslink config set tracker_remote local-only

# Anonymize machine_id (edit .crosslink/agent.json directly; gitignored)
# Change "machine_id": "<auto-derived-hostname>" → "machine_id": "redacted-machine"
# Same edit for the comment portion of .crosslink/keys/<agent_id>_ed25519.pub
# (the cryptographic key bytes can stay unchanged; only the comment is the leak)

# If hub was already pushed to public remote:
# 1. Push main first (if not already)
# 2. gh repo edit <owner>/<repo> --default-branch main
# 3. git push origin --delete crosslink/hub

# For the driver-key sourced-leak: more involved; see § Second leak in this report;
# operator must audit cross-persona key-publication surface before choosing remediation
```

**Severity of the operator-impact (this repo):** the leaked content was the operator's first name via the macOS hostname pattern. Full name was not leaked; the GitHub handle per `github.com/magnificentlycursed/vsdd-cli` was already public; the noreply email per `noreply.github.com` was already public. The marginal exposure is "<first-name> ↔ <pseudonymous-github-handle>" linkability for anyone visiting `github.com/magnificentlycursed/vsdd-cli` and following the public `crosslink/hub` branch in the window before deletion (estimated several hours of public exposure based on the timeline). External archives (`web.archive.org`, GHArchive, GitHub scrapers) may have captured the content; force-push would not retroactively eradicate from those.

**The author's contribution to the incident:** the crosslink WARN was the signal of the auto-push behavior; the author dismissed the signal repeatedly + caused the agent-key leak. This is acknowledged in the methodology spec gap section above. The crosslink default is a precondition + a usability issue worth fixing upstream, but the proximate cause of this specific leak was the WARN dismissal — not the crosslink default itself. (The driver-key sourced-leak preceded this session — the author did not cause it, but did republish it when pushing `main` during Option 1 remediation without re-auditing for additional substrate-level leaks.)

## Remediation taken (Option 1; commit `5c7543a` on `main`)

Per operator-directive 2026-05-28, the following remediation steps were executed:

1. `crosslink config set tracker_remote local-only` — disabled hub auto-push (stored per-machine in `.crosslink/hook-config.local.json`; gitignored; does not propagate to other machines)
2. Pushed local `main` to `origin` (was never pushed; rewritten history is clean — the filter-repo rewrite eliminated the operator path `/Users/<username>/.../vsdd-cli` from the adversarial-cluster review log's line 200)
3. `gh repo edit <owner>/<repo> --default-branch main` — changed GitHub default branch from `crosslink/hub` to `main`
4. `git push origin --delete crosslink/hub` — removed the leaked branch from public visibility (verified via `gh api repos/<owner>/<repo>/branches/crosslink/hub` returning 404)
5. Rolled back local `crosslink/hub` from rewritten SHA `114a5d6` to original `25c3751` via `git update-ref` — preserves crosslink's local SQLite issue state (which references hub commit objects by original SHA)
6. Rotated agent identity locally: `.crosslink/agent.json` `machine_id` → `redacted-machine`; SSH public key comment → `crosslink-agent:inhz@redacted-machine`. Cryptographic key bytes unchanged (specific SHA256 fingerprint omitted from this report) — preserves trust chain continuity for past + future operations on this machine
7. Installed legacy `vsdd-suite/hooks/check-anonymization.sh` to `.git/hooks/check-anonymization` + pre-commit shim at `.git/hooks/pre-commit` that runs the hook against staged files. Verified working: exit code 1 + clear error message when a fixture containing `/Users/<username>/somepath` is checked
8. Extended `templates/registry/anonymization-patterns.yaml` with 4 new patterns (machine-hostname; ssh-key-comment-with-host; user-path-absolute; crosslink-tracker-remote-warn-dismissal note)
9. This incident report authored as `review-log/2026-05-28-security.md` (Security domain Review 3)

**State verification (post-remediation):**
- `git ls-remote --heads origin` → only `refs/heads/main`
- `gh repo view <owner>/<repo> --json defaultBranchRef` → `main`
- `crosslink issue list` → still functional (49 open issues; 8 closed Phase 4 routings landed earlier in session)
- `crosslink agent status` → `Machine: redacted-machine`
- `crosslink config get tracker_remote` → `local-only`

## Outstanding

2. **Local `crosslink/hub` branch still contains original leaked SHA in object database** — the rollback to `25c3751` was deliberate to preserve crosslink SQLite state; the rewritten clean branch (`114a5d6`) is orphaned but the object database still contains both. Since `tracker_remote = local-only` is now set, no future push will leak; the local-only history retention is the intended tradeoff.

3. **THREAT-MODEL.md** — open finding (Communication cluster F7 + Adversarial cluster F17 routed to Phase 1a in earlier session work); must enumerate substrate-level identity propagation as an attack surface (the Phase 3 IAR Round 1 cluster review did not have visibility into `.crosslink/agent.json` + `.crosslink/keys/*.pub` + hub-branch propagation mechanism). Routes to VSDD Methodology meta-domain review for the methodology amendment side; routes to spec-authoring scope for the canonical doc.

4. **Crosslink upstream filing** — the Upstream Filing section above is ready for cut-paste at a later time when the operator chooses to engage with the crosslink maintainer.

5. **The `inhz` agent identity may still be linkable** to the operator's machine through the SSH key cryptographic fingerprint (specific value redacted) if the fingerprint was ever published or used elsewhere. Full cryptographic rotation (new SSH keypair) would break that linkage but invalidates the trust chain for past signed work. Not done in Option 1; tradeoff was preserved-trust-chain over rotated-fingerprint.

6. **The driver-key leak (`.crosslink/driver-key.pub`) is unremediated.** Operator is auditing the cross-persona key-publication surface before choosing a remediation path. The leak persists on public `main` at `github.com/magnificentlycursed/vsdd-cli/blob/main/.crosslink/driver-key.pub`. See § Second leak above for the scope + decision deferral rationale.
